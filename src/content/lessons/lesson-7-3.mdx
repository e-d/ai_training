## Why Traditional Databases Fall Short

When your LLM application needs to find relevant information — a customer asking about a return policy, an analyst searching internal research, a support agent looking for a similar past ticket — it needs to search by meaning, not by exact keywords. Traditional databases are designed for exact matches and structured queries: "find all orders where status = 'shipped' and date > '2024-01-01'." They have no concept of semantic similarity.

Vector databases solve this problem. They store information as mathematical vectors — arrays of numbers generated by embedding models — and retrieve results based on how mathematically close those vectors are to a query. This is the technology that makes it possible for an LLM application to understand that a question about "firing employees" is related to a document about "workforce reduction procedures" even though they share no keywords.

## How Vector Search Works

The process has three stages, and none of them require the business leader to understand linear algebra — only the logic of what's happening:

### Stage 1: Ingestion

Your documents — policies, product descriptions, research reports, support tickets, whatever your application needs access to — are split into chunks (typically a few paragraphs each) and run through an embedding model. The embedding model converts each chunk into a vector: an array of numbers (commonly 768 to 3,072 dimensions) that encodes the semantic meaning of that text. These vectors are stored in the vector database alongside metadata like the source document, date, and category.

### Stage 2: Indexing

The vector database organizes these vectors using specialized data structures that enable fast similarity search across millions or billions of vectors. The most common approach is Approximate Nearest Neighbor (ANN) search, which trades a tiny amount of accuracy for massive speed gains. Pinecone, a leading vector database, reports query latencies under 50 milliseconds on indexes containing over a billion vectors.

### Stage 3: Querying

When a user asks a question, the system converts the question into a vector using the same embedding model, then asks the vector database: "What stored vectors are most similar to this query vector?" The database returns the top-K most similar chunks, which are then passed to the LLM as context for generating an answer. This is the retrieval step in Retrieval-Augmented Generation.

## The Vector Database Market

The vector database market has grown rapidly. Grand View Research valued it at $1.5 billion in 2023 and projects it to reach $4.3 billion by 2028, driven almost entirely by LLM application demand. Several distinct categories of products compete:

### Purpose-Built Vector Databases

**Pinecone** is the most widely adopted fully managed vector database. It offers a serverless architecture where customers pay per query rather than provisioning infrastructure. Pinecone handles indexing, scaling, and replication automatically — which appeals to organizations that want vector search without hiring specialized infrastructure engineers. Companies including Shopify, Hubspot, and Notion use Pinecone in production.

**Weaviate** differentiates itself with built-in vectorization — it can generate embeddings from text, images, and other data types automatically, without requiring a separate embedding model call. Weaviate is open-source with a managed cloud offering, giving organizations the option to self-host for data sovereignty.

**Qdrant** has gained traction in performance-sensitive applications. Written in Rust, it delivers high query throughput with low resource consumption. Qdrant's filtering capabilities — the ability to combine vector similarity search with traditional metadata filters — make it popular for applications that need both semantic and structured search.

### Established Databases Adding Vector Support

The major database vendors have added vector capabilities to their existing products:

- **PostgreSQL** gained vector search through the pgvector extension, which lets organizations add semantic search to their existing PostgreSQL databases without adopting new infrastructure. For many companies, this "good enough" approach avoids the complexity of a separate vector database.
- **Elasticsearch** added dense vector search, enabling organizations already using Elasticsearch for text search to add semantic capabilities.
- **MongoDB Atlas** introduced vector search in 2023, letting MongoDB users store vectors alongside their existing document data.
- **Redis** added vector similarity search to its in-memory data store, offering extremely low latency for applications where speed is critical.

### Which Category Wins?

The choice depends on your starting point and requirements. If you're building a new LLM application from scratch and need the best retrieval quality, a purpose-built vector database like Pinecone or Weaviate is the strongest option. If you already run PostgreSQL and need "good enough" vector search without adding another database to your infrastructure, pgvector is a pragmatic choice. A 2024 benchmark by Zilliz found that purpose-built vector databases outperformed add-on solutions by 2-5x on query latency and throughput — but for many use cases, the add-on performance is more than sufficient.

## Chunking: The Unsung Hero of Retrieval Quality

The most underappreciated factor in vector search quality isn't the database — it's how you chunk your documents before embedding them. Chunking is the process of splitting documents into pieces that will be individually stored and retrieved.

If chunks are too large, the embedding captures a vague average of too many topics, and retrieval becomes imprecise. If chunks are too small, individual chunks lack enough context to be useful. The sweet spot depends on your content: structured documents like policies might be chunked by section headings, while narrative text might use overlapping windows of 500-1,000 tokens.

Advanced chunking strategies include:

- **Semantic chunking** — using an LLM to identify natural topic boundaries rather than splitting at fixed character counts
- **Hierarchical chunking** — storing both summary-level and detail-level chunks, searching summaries first, then drilling into details
- **Parent-child chunking** — retrieving a small, precise chunk but passing the larger parent section to the LLM for context

Organizations that invest in chunking strategy consistently see 20-40% improvements in retrieval quality over naive fixed-size chunking, according to a 2024 analysis by LlamaIndex.

## Real-World Vector Database Deployments

**Spotify** uses vector embeddings to power podcast search and recommendations. Each podcast episode is embedded based on its transcript, description, and metadata, enabling users to find content by describing what they want to learn rather than knowing the exact title.

**Instacart** deployed vector search to power its "Ask Instacart" feature, which lets customers describe meals or dietary needs in natural language and receive personalized product recommendations from their local store's inventory — combining vector similarity with real-time availability data.

**Notion** uses vector search to power its Q&A feature, which lets users ask questions about their workspace content and receive answers grounded in their own documents, wikis, and notes.

> **Key takeaway:** Vector databases are the memory layer of LLM applications — they enable AI systems to search by meaning rather than keywords, grounding model responses in your organization's actual data. The quality of your vector search depends less on which database you choose and more on how well you prepare and chunk your data before storing it.
